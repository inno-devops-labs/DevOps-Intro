I.

AWS — Amazon ECR (Elastic Container Registry)
Это основной сервис AWS для хранения контейнерных образов. 
Поддерживает Docker-образы и OCI-форматы. 

Ключевые особенности: есть встроенное сканирование уязвимостей. Интегрируется с ECS, EKS, Lambda и CodePipeline. 
По цене: платишь за объем хранилища ($0.10 за ГБ в месяц) и за передачу данных наружу.
Часто используют для хранения образов микросервисов и CI/CD конвейеров.

Также у AWS есть CodeArtifact для пакетов (npm, PyPI, Maven, NuGet), но он считается отдельным сервисом, не основным реестром.

GCP — Artifact Registry
Официальное название — Google Artifact Registry. 
В отличие от старого Container Registry, этот универсальный: поддерживает не только контейнеры (Docker и OCI), но и пакеты вроде Java (Maven), JavaScript (npm), Python, Go и даже Debian-пакеты. 
Ключевые функции: автоматическое сканирование уязвимостей, настройка политик удаления, репликация между регионами, а также тонкий контроль доступа через IAM. 
Отлично интегрируется с Cloud Run, GKE, Cloud Build и даже с Pub/Sub для событий. 
Цена: $0.10 за ГБ в месяц для хранения, передача трафика внутри GCP бесплатная, но исходящий наружу — платный.
Используют как единое хранилище для артефактов в пайплайнах разработки.

Azure — Azure Container Registry (ACR)
Основной сервис от Microsoft. 
Поддерживает Docker-образы и OCI-артефакты (например, Helm-чарты, CNAB). 
Из важных фич: георепликация (доступна на Premium-уровне), встроенное сканирование уязвимостей через Defender for Cloud, а также возможность задания политик и подписей доверенных образов (Content Trust). 
Интегрируется с AKS, App Service, Azure Pipelines и даже с Azure Red Hat OpenShift. 
Часто используют для хранения образов приложений, которые деплоятся в Kubernetes.

Сравнение и анализ для мультиоблачной стратегии
Все три сервиса решают похожие задачи, но с нюансами. 

AWS держит контейнеры отдельно (ECR), а пакеты - в CodeArtifact. 
GCP же объединил всё в Artifact Registry, что удобнее для мультиязычных проектов. 
Azure ACR тоже достаточно универсален, но упор на контейнеры и OCI-артефакты.

II.

Сервисы по провайдерам
AWS - AWS Lambda. Запускает код на события, масштабируется автоматически.

GCP - Cloud Run (сейчас он основной) и Cloud Functions. Cloud Run работает с контейнерами и масштабируется до нуля, что удобно.

Azure - Azure Functions. Главная фишка - разные планы хостинга (Consumption, Premium) и Durable Functions для длительных процессов.

Ключевые характеристики
По языкам: везде есть Python, Node.js, Java. У Azure ещё C# из коробки. Lambda поддерживает Go, Ruby.

Ограничения: Lambda максимум 15 минут работы, Cloud Run до 60 минут (на втором поколении функций), Azure Functions на Premium плане без ограничений.

Холодные старты: у всех есть. Самые быстрые на Cloud Run (легковесные контейнеры). Lambda средняя, Azure на Consumption может быть медленнее.

Выбор для REST API
Судя по ответу Дипсика))), я бы взял AWS Lambda. Почему: у AWS самый большой опыт в serverless, много готовых примеров. Для простого API это самый надёжный вариант.

Плюсы и минусы serverless
Плюсы: не нужно администрировать серверы, платишь только за реальное использование, автоматическое масштабирование. Разработка быстрее.
Минусы: холодные старты, жёсткие ограничения по времени (не для всех задач), при высокой нагрузке может выходить дороже обычных серверов. И привязка к конкретному облаку.
