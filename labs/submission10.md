# Лабораторная работа 10 - Основы облачных вычислений

## Задание 1 - Исследование реестров артефактов

### AWS

У AWS нет одного полностью универсального artifact registry, как у Google Cloud. На практике эту роль делят два сервиса: Amazon Elastic Container Registry (Amazon ECR) и AWS CodeArtifact.

Amazon ECR - это основной управляемый registry для контейнерных артефактов. Он хранит Docker-образы, OCI-образы и OCI-совместимые артефакты, поддерживает IAM, политики репозиториев, сканирование образов, межрегиональную и межаккаунтную репликацию, а также pull-through cache.

AWS CodeArtifact дополняет ECR и предназначен уже не для контейнеров, а для пакетов. Он поддерживает Cargo, generic packages, Maven, npm, NuGet, PyPI, Ruby и Swift. По модели оплаты оба сервиса работают по модели pay-as-you-go: ECR берет оплату в основном за хранение и сетевой трафик, а CodeArtifact - за хранение, запросы и исходящий трафик.

Из этого видно, что у AWS используется разделенная модель registry. Для контейнеров отлично подходит ECR, а для внутренних репозиториев пакетов - CodeArtifact. Внутри AWS такой подход работает хорошо, но он менее цельный, чем единый сервис, который хранит и контейнеры, и пакеты одновременно.

### GCP

Google Artifact Registry в этом сравнении выглядит как наиболее универсальный сервис. Google позиционирует его как единое место для хранения и управления пакетами и контейнерными образами. Он напрямую интегрируется с Cloud Build, GKE, Cloud Run, Compute Engine и App Engine.

Artifact Registry поддерживает широкий набор форматов: Docker/OCI, Helm в OCI-формате, Maven, npm, Python, Apt, Yum, шаблоны Kubeflow Pipelines, Go modules и generic artifacts. Для безопасности и управления доступны IAM и сканирование уязвимостей через Artifact Analysis.

Сильная сторона сервиса - модель репозиториев:

- standard repositories для приватных артефактов;
- remote repositories для кэширования внешних источников вроде Docker Hub, Maven Central и PyPI;
- virtual repositories для единой точки доступа сразу к нескольким репозиториям.

Оплата строится на хранении, передаче данных и дополнительных возможностях вроде сканирования уязвимостей.

С архитектурной точки зрения Google Artifact Registry удобен тем, что объединяет управление пакетами, контроль зависимостей, CI/CD-интеграции и элементы безопасности цепочки поставки в одном продукте. Особенно полезны remote и virtual repositories, потому что они упрощают контроль внешних зависимостей.

### Azure

Azure Container Registry (ACR) - это управляемый приватный registry от Microsoft для контейнерных образов и связанных OCI-артефактов. Он поддерживает Docker-совместимые образы, Helm charts, OCI artifacts, а также артефакты цепочки поставки, например подписи, SBOM и результаты сканирования.

ACR поддерживает автоматизацию сборки через ACR Tasks и хорошо интегрируется с AKS, Azure App Service, Azure Machine Learning, Azure Batch и Azure Red Hat OpenShift. В плане сети и геораспределения сильнее всего выглядит тариф Premium, потому что именно он добавляет георепликацию и Private Link/private endpoints.

У Azure модель оплаты построена на уровнях Basic, Standard и Premium. Они отличаются по объему включенного хранилища, пропускной способности и набору функций. Для оценки уязвимостей ACR может интегрироваться с Microsoft Defender for Cloud.

По сравнению с Google Artifact Registry, ACR больше сфокусирован именно на OCI- и контейнерных сценариях, а не на универсальном управлении пакетами. По сравнению с AWS он выглядит более цельным для контейнерных нагрузок, но все равно покрывает экосистемы языковых пакетов слабее, чем Google Artifact Registry.

### Анализ: какой сервис лучше для мультиоблачной стратегии

Если бы мне нужно было выбрать один вариант для мультиоблачной стратегии, я бы выбрал Google Artifact Registry. Главная причина в том, что у Google самый цельный и единый подход: один сервис покрывает контейнерные образы, языковые пакеты, системные пакеты, generic artifacts и дополнительные сценарии вроде шаблонов ML-пайплайнов. Кроме того, remote и virtual repositories позволяют централизовать управление зависимостями.

AWS очень силен, но его подход более фрагментирован, потому что для полной картины часто нужны и ECR, и CodeArtifact. Azure Container Registry отлично подходит для enterprise-сценариев с OCI-контейнерами, особенно если важны георепликация, приватная сеть и тесная интеграция с Azure. Но как общий мультиоблачный registry для разных типов артефактов Google Artifact Registry выглядит гибче и удобнее.

## Задание 2 - Исследование бессерверных платформ

### AWS

AWS Lambda - основная serverless compute-платформа AWS. Она запускает код без управления серверами, автоматически масштабируется и использует модель оплаты за фактическое использование. Lambda поддерживает управляемые рантаймы, запуск через контейнерные образы и пользовательские рантаймы.

Lambda очень тесно связана с event-driven архитектурой AWS: ее можно запускать от SQS, SNS, EventBridge, DynamoDB Streams, S3 и многих других сервисов. Для HTTP API Lambda обычно используется вместе с Amazon API Gateway. Оплата идет за количество вызовов и время выполнения. Максимальный timeout составляет 900 секунд (15 минут).

Lambda особенно сильна в event-driven сценариях внутри AWS. Для REST API она тоже подходит, но ограничения по времени выполнения и привязка к модели "функция + API Gateway" делают ее немного менее гибкой, чем контейнерно-ориентированный подход Cloud Run.

### GCP

Cloud Run - основная serverless compute-платформа Google Cloud. Google описывает ее как полностью управляемую платформу, которая запускает код, функции и контейнеры на масштабируемой инфраструктуре Google. Важный момент в том, что Cloud Run не ограничивается только классическими функциями: он может запускать полноценные контейнеры, сервисы из исходного кода и Cloud Run functions.

Cloud Run поддерживает HTTP-запросы как основной сценарий, а event-driven запуск работает через Eventarc. Для функций на Cloud Run поддерживаются Node.js, Python, Go, Java, Ruby, PHP и .NET. Для HTTP-сценариев timeout может достигать 60 минут, а concurrency - до 1000 запросов на инстанс. Модель оплаты может быть по запросам или по инстансам.

Платформа сильна тем, что объединяет serverless-подход и гибкость контейнеров. Поэтому она хорошо подходит для REST API, webhook-сервисов, background handlers и других приложений, которые уже живут в контейнерной модели.

### Azure

Azure Functions - event-driven serverless-платформа Microsoft. Она подходит для web API, workflow-сценариев, обработки очередей, реакций на изменения в БД и других интеграционных задач. Azure Functions поддерживает C#, Java, JavaScript, PowerShell и Python, а также пользовательские handlers для других языков, например Rust или Go.

Особенность Azure Functions - модель triggers and bindings, которая уменьшает количество шаблонного кода при интеграции с сервисами Azure. Microsoft рекомендует Flex Consumption для новых serverless-приложений с динамическим масштабированием. Этот план уменьшает cold starts, поддерживает интеграцию с виртуальной сетью, масштабирование на уровне отдельных функций и настраиваемую concurrency. План Premium добавляет always-ready и prewarmed instances для еще более стабильного отклика.

Timeout зависит от плана: у классического Consumption по умолчанию 5 минут и максимум 10 минут, а у Flex Consumption и Premium по умолчанию 30 минут и практически неограниченное выполнение с учетом ограничений платформы. На динамических планах оплата идет за вызовы и потребленные ресурсы, а Premium оплачивается по выделенным ресурсам.

### Сравнительная таблица serverless-платформ

| Провайдер | Основной сервис | Модель выполнения | Языки и рантаймы | Масштабирование и лимиты | Модель оплаты |
| --------- | --------------- | ----------------- | ---------------- | ------------------------ | ------------- |
| AWS | AWS Lambda | Функции, event-driven подход. HTTP обычно через API Gateway | Управляемые рантаймы, контейнерные образы, пользовательские рантаймы | Автомасштабирование, cold starts возможны, максимум 15 минут | Запросы + время выполнения |
| GCP | Cloud Run | Serverless для HTTP и событий, container-native подход | Контейнеры произвольного состава; функции на Node.js, Python, Go, Java, Ruby, PHP, .NET | Scale to zero, minimum instances, до 60 минут для HTTP, высокая concurrency | Оплата по запросам или по инстансам |
| Azure | Azure Functions | Event-driven функции с triggers and bindings | C#, Java, JavaScript, PowerShell, Python, пользовательские handlers | Flex снижает cold starts, Premium дает warm instances, Consumption максимум 10 минут | Consumption/Flex: вызовы + ресурсы; Premium: выделенные ресурсы |

### Анализ: что лучше для backend под REST API

Для backend под REST API я бы выбрал Google Cloud Run. На мой взгляд, его главное преимущество в том, что HTTP-нагрузка для него является одним из основных сценариев, а не просто дополнительным способом вызвать функцию. Cloud Run хорошо подходит для сервисов, которые уже собираются в контейнеры, поддерживает долгие HTTP-запросы, высокую concurrency и две модели оплаты.

AWS Lambda я бы поставил на второе место. Это очень зрелая и надежная платформа, особенно если backend уже глубоко встроен в AWS-экосистему. Но ограничение в 15 минут и более "функциональный" стиль работы делают ее чуть менее универсальной для общего случая REST API, чем Cloud Run.

Azure Functions я бы поставил на третье место для общего случая. Не потому что сервис слабый, а потому что при выборе нужно сильнее разбираться в различиях между Consumption, Flex Consumption и Premium. Если инфраструктура компании уже сильно завязана на Azure, то Azure Functions тоже может быть очень хорошим выбором.

## Reflection - Преимущества и недостатки бессерверных вычислений

Главные преимущества serverless-подхода хорошо видны у всех трех провайдеров. Во-первых, не нужно управлять серверами, а значит снижается операционная нагрузка. Во-вторых, платформы автоматически масштабируются под нагрузку. В-третьих, оплата ближе к реальному использованию, что особенно выгодно для небольших сервисов и непостоянной нагрузки.

Недостатки тоже очевидны. Serverless-платформы могут давать cold starts, особенно когда приложение масштабируется с нуля. Кроме того, у них есть ограничения по среде выполнения и timeout, причем у разных провайдеров и тарифных планов они различаются. Еще один минус - это более сильная зависимость от конкретного облака, потому что приложение начинает использовать облачные triggers, identity-модели и специфичные deployment workflows.

На мой взгляд, serverless лучше всего подходит для API, webhook-сервисов, scheduled jobs, обработчиков очередей, integration services и event processing. Хуже он подходит для долгих синхронных задач, сценариев с очень тонким контролем среды выполнения и случаев, где нужны предсказуемые always-on ресурсы.
