\# Лабораторная работа №10: Cloud Computing Fundamentals



\## Задание 1 – Артефакт‑реестры



\### AWS – Elastic Container Registry (ECR)

\- Полностью управляемый реестр контейнеров Docker/OCI.

\- Поддерживает сканирование уязвимостей, политики жизненного цикла, репликацию между регионами.

\- Интегрируется с ECS, EKS, CodePipeline, а также с любыми CI/CD через стандартный Docker CLI.

\- \*\*Цена:\*\* \~$0.10 за ГБ/мес хранения + плата за исходящий трафик.



\### GCP – Artifact Registry (GAR)

\- Универсальный реестр: хранит не только контейнеры, но и пакеты Maven, npm, PyPI, Go, apt, yum.

\- Встроенные политики очистки, автоматическое шифрование, синхронизация с внешними реестрами.

\- Глубоко интегрирован с GKE, Cloud Run, Cloud Build.

\- \*\*Цена:\*\* аналогична AWS (\~$0.10/ГБ/мес).



\### Azure – Container Registry (ACR)

\- Реестр контейнеров с поддержкой OCI‑образов и Helm‑чартов.

\- Георепликация, встроенная задача ACR Tasks (сборка и исправление образов в облаке).

\- Сканирование уязвимостей через Microsoft Defender for Cloud.

\- \*\*Цена:\*\* также около $0.10/ГБ/мес.



\### Сравнение



| Провайдер | Сервис | Форматы | Ключевая фишка |

|-----------|--------|---------|----------------|

| AWS | ECR | Docker, OCI, Helm | IAM + репликация |

| GCP | Artifact Registry | Docker, Helm, Maven, npm, PyPI, Go, apt, yum | Универсальность |

| Azure | ACR | Docker, OCI, Helm | ACR Tasks, георепликация |



\*\*Для multi‑cloud стратегии\*\* лучше выбрать \*\*GCP Artifact Registry\*\* – он покрывает почти все типы артефактов, что позволяет унифицировать хранение в гибридной среде.

\## Задание 2 – Serverless платформы



\### AWS Lambda

\- Запуск кода по событиям (HTTP, очередь, S3, таймер).

\- Языки: Node, Python, Java, Go, Ruby, .NET, Rust.

\- Макс. память: 10 ГБ, время выполнения: 15 минут.

\- Холодный старт: \~100–300 мс.

\- Бесплатно: 1 млн запросов + 400 000 ГБ·с в месяц.

\- Платно: $0.20/млн запросов + $0.0000166667/ГБ·с.



\### GCP Cloud Run

\- Запуск любого HTTP‑контейнера в serverless‑режиме (Knative).

\- Поддерживает любой язык (что поместится в контейнер).

\- Макс. память: 8 ГБ, время выполнения: до 60 минут.

\- Холодный старт: \~200–400 мс (можно держать 1 экземпляр постоянно).

\- Бесплатно: 2 млн запросов + 180 000 ГБ·с + 180 000 vCPU·с.

\- Платно: $0.40/млн запросов + $0.000024/ГБ·с + $0.000012/vCPU·с.



\### Azure Functions

\- Событийная модель с множеством триггеров (HTTP, таймер, Blob, Queue).

\- Языки: C#, Node, Python, Java, PowerShell, пользовательские обработчики.

\- Consumption plan: память 1.5 ГБ, время 5 минут, холодный старт \~0.5–1 с.

\- Бесплатно: 1 млн запросов + 400 000 ГБ·с.

\- Платно: $0.20/млн запросов + $0.000016/ГБ·с.



\### Сравнение



| Провайдер | Сервис | Модель | Макс. память | Макс. время | Холодный старт |

|-----------|--------|--------|--------------|-------------|----------------|

| AWS | Lambda | событийная | 10 ГБ | 15 мин | \~100–300 мс |

| GCP | Cloud Run | контейнерная | 8 ГБ | 60 мин | \~200–400 мс |

| Azure | Functions | событийная | 1.5 ГБ | 5 мин | \~500–1000 мс |



\*\*Для REST API бэкенда\*\* я бы выбрал \*\*GCP Cloud Run\*\*:

\- Можно использовать готовое приложение в контейнере (любой фреймворк).

\- Долгое время выполнения (60 мин) подходит для тяжёлых запросов.

\- Гибкое управление холодным стартом (min instances).



\### Плюсы serverless

\- Нет администрирования серверов.

\- Автоматическое масштабирование от 0 до тысяч.

\- Оплата только за реальное использование.



\### Минусы serverless

\- Холодный старт (задержка при первом вызове).

\- Ограничения по памяти, времени, размеру кода.

\- Сложнее отладка и локальное тестирование.

\- Привязка к конкретному облачному провайдеру (вендор‑локин).
