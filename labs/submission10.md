# Lab 10 — Основы облачных вычислений

## Задание 1 — Реестры артефактов

Реестры артефактов — это хранилища для контейнерных образов, пакетов и других
артефактов сборки. Они являются важной частью CI/CD пайплайнов, потому что позволяют
хранить, версионировать и раздавать артефакты между командами и окружениями.
Без реестра каждый раз пришлось бы собирать образы с нуля или передавать их вручную.

### AWS — Amazon Elastic Container Registry (ECR)

Основной реестр у AWS — это Amazon ECR. Он хранит Docker и OCI-образы, а также
Helm-чарты. Для пакетов вроде npm или Maven у AWS есть отдельный сервис CodeArtifact,
то есть контейнеры и пакеты хранятся в разных местах.

Из ключевых возможностей ECR стоит выделить следующее:
- Сканирование уязвимостей через Amazon Inspector — работает автоматически при пуше
- Репликация образов между регионами — настраивается правилами на уровне реестра
- Неизменяемые теги (immutable tags) — нельзя перезаписать тег, что делает деплои
  предсказуемыми
- Публичный реестр ECR Public Gallery — можно публиковать образы как на Docker Hub
- Приватные эндпоинты через VPC — трафик не выходит в интернет
- Доступ управляется через AWS IAM

Цена: хранилище стоит $0.10 за гигабайт в месяц, трафик внутри AWS бесплатный.
В рамках Free Tier первые 500 МБ в месяц бесплатны.

ECR чаще всего используют для хранения образов приложений, которые деплоятся в ECS
или EKS, а также для снабжения образами CodeBuild в CI/CD пайплайнах.

### GCP — Artifact Registry

У Google Cloud основной реестр — Artifact Registry. Он пришёл на замену старому
Container Registry и сильно его превосходит по возможностям. Главное отличие от
конкурентов — поддержка множества форматов в одном сервисе:
- Docker и OCI-образы
- Maven (Java)
- npm (Node.js)
- Python (PyPI)
- Go-модули
- Debian и RPM-пакеты
- Helm-чарты

Это значит, что не нужен отдельный сервис для пакетов — всё хранится в одном месте.

Из интересных возможностей:
- Региональные и мультирегиональные репозитории — можно выбрать, где хранятся данные
- CMEK (ключи шифрования, управляемые клиентом) — настраивается на уровне каждого
  репозитория отдельно, а не на уровне аккаунта целиком
- Remote Repositories — Artifact Registry может работать как прокси для Docker Hub,
  PyPI и Maven Central, кешируя пакеты локально. Это ускоряет сборки и защищает от
  недоступности upstream-реестров
- Сканирование уязвимостей через Container Analysis
- Трафик в GKE и Cloud Run внутри того же региона — бесплатный

Цена: $0.10 за ГБ в месяц, 500 МБ бесплатно каждый месяц на постоянной основе (не
только в первый год).

### Azure — Azure Container Registry (ACR)

У Azure основной реестр для контейнеров — это Azure Container Registry. Для пакетов
(npm, Maven, NuGet) используется отдельный сервис Azure Artifacts в составе Azure DevOps.

ACR поддерживает Docker, OCI-образы, Helm-чарты, а также WASM-модули и Bicep-файлы.
Из особенностей:
- Три уровня (SKU): Basic (~$5/мес), Standard (~$20/мес), Premium (~$50/мес)
- Гео-репликация и Private Link доступны только на Premium
- ACR Tasks — автоматическая пересборка образов при обновлении базового образа или
  при коммите в репозиторий
- Сканирование через Microsoft Defender for Containers
- Глубокая интеграция с Azure Active Directory и Azure DevOps

Цена: фиксированная месячная плата за SKU, бесплатного уровня нет — минимум $5/мес.

### Сравнение реестров

Если сравнивать три сервиса, можно выделить следующее:
- По форматам артефактов GCP Artifact Registry выигрывает — он поддерживает Maven,
  npm, PyPI и другие прямо из коробки, без второго сервиса
- По цене AWS и GCP примерно одинаковы (pay-per-use), Azure дороже из-за фиксированного
  SKU и отсутствия бесплатного уровня
- Гео-репликация есть у всех, но у Azure она только на Premium
- Сканирование уязвимостей есть у всех трёх
- Remote Repositories (кеш upstream-реестров) — только у GCP
- AWS ECR жёстко завязан на AWS IAM, что немного усложняет интеграцию с внешними
  CI-системами (GitLab, GitHub Actions требуют настройки OIDC)

### Что выбрать для мультиоблачной стратегии

Для мультиоблачной стратегии я бы выбрала GCP Artifact Registry. Основная причина —
это единственный сервис среди трёх, который закрывает все форматы артефактов: и
контейнеры, и пакеты для Java, Python, Node.js и других языков. В мультиоблачном
окружении меньше зависимостей от конкретного провайдера — это плюс.

Дополнительно Remote Repositories решают реальную проблему: в крупных командах часто
нужен надёжный кеш для PyPI и Docker Hub, чтобы сборки не падали из-за недоступности
внешних реестров. У AWS и Azure такого нет. Плюс CMEK на уровне репозитория и
прозрачное ценообразование без обязательной подписки делают его удобнее для проектов
с разными командами с разными требованиями к безопасности.

Azure ACR имеет смысл, если вся инфраструктура уже на Azure и активно используется
Azure AD — тогда интеграция нативная и удобная. Но для мультиоблачного сценария это
слишком завязано на экосистему Microsoft.



## Задание 2 — Serverless-платформы

Serverless — это модель, при которой разработчик деплоит функции или контейнеры, а
провайдер сам управляет серверами, масштабированием и доступностью. Оплата идёт
только за реальное время выполнения — если функция не вызывается, она ничего не
стоит. Это избавляет от необходимости держать постоянно работающие серверы для
непредсказуемой нагрузки.

### AWS — AWS Lambda

Lambda — самый зрелый serverless-сервис на рынке. Поддерживает Node.js, Python, Java,
Go, Ruby, .NET, а через Custom Runtimes можно запустить что угодно, включая Rust.

Ключевые параметры:
- Максимальное время выполнения: 15 минут
- Память: от 128 МБ до 10 240 МБ (CPU масштабируется пропорционально памяти)
- Конкурентность по умолчанию: 1000 одновременных вызовов (можно увеличить)
- Триггеры: 200+ сервисов AWS — S3, SQS, DynamoDB Streams, API Gateway, EventBridge
  и другие
- Поддержка контейнерных образов до 10 ГБ

Cold start — одна из проблем Lambda: Python и Node.js стартуют за 100–300 мс, Java и
.NET могут стартовать за 1–2 секунды. Для production API есть Provisioned Concurrency —
инстансы держатся прогретыми за дополнительную плату.

Цена: 1 млн вызовов и 400 000 ГБ-секунд в месяц бесплатно на постоянной основе. Далее
$0.20 за 1 млн запросов и $0.0000166 за ГБ-секунду.

### GCP — Google Cloud Functions

У Google основной serverless-сервис для функций — Cloud Functions, сейчас актуальна
2-я версия. Она построена поверх Cloud Run, что даёт важное преимущество: один инстанс
может обрабатывать до 1000 запросов одновременно (в отличие от Lambda, где каждый
инстанс — один запрос). Это резко снижает количество cold start'ов при нагрузке.

Поддерживаемые языки: Node.js, Python, Go, Java, Ruby, PHP, .NET.

Ключевые параметры:
- Максимальное время выполнения: 60 минут для HTTP-функций (в 4 раза больше, чем у Lambda)
- Память: от 128 МБ до 32 768 МБ
- Cold start: 200 мс – 1.5 с
- Триггеры: HTTP, Pub/Sub, Cloud Storage, Firestore, Eventarc

Цена: 2 млн вызовов в месяц бесплатно (вдвое больше, чем у AWS и Azure). Далее по
pay-per-use. Трафик в GKE и Cloud Run в том же регионе бесплатный.

### Azure — Azure Functions

Azure Functions поддерживает C#, Java, Python, Node.js, PowerShell и Custom Handlers
для любого другого языка.

Самая интересная особенность Azure Functions — это Durable Functions: расширение для
написания stateful-оркестраций прямо в коде функций. Это позволяет реализовывать
сложные многошаговые воркфлоу (например, процесс согласования с ожиданием ответа
человека) без внешних сервисов. У Lambda и Cloud Functions такого нет из коробки.

Планы хостинга:
- Consumption (pay-per-use): максимум 10 минут, память до 1.5 ГБ
- Flex Consumption: максимум 60 минут
- Premium: без лимита по времени, прогретые инстансы, но фиксированная цена ~$150/мес
- Dedicated: запуск на App Service, полный контроль

Cold start на Consumption — 200 мс – 3 с, самый медленный из трёх провайдеров. На
Premium этой проблемы нет.

Цена: 1 млн вызовов и 400 000 ГБ-секунд бесплатно в месяц. Далее $0.20 за 1 млн
запросов и $0.000016 за ГБ-секунду.

### Сравнение serverless-платформ

Если сравнивать по основным параметрам:
- По максимальному времени выполнения лидирует GCP (60 мин), Lambda ограничена 15 мин,
  Azure Consumption — 10 мин (но Premium убирает лимит)
- По памяти GCP даёт больше всего — до 32 ГБ, Lambda до 10 ГБ, Azure Consumption
  только 1.5 ГБ — это ощутимое ограничение
- По cold start быстрее всего Lambda на Python/Node.js (100 мс), GCP в среднем (200 мс),
  Azure медленнее (до 3 с)
- По количеству бесплатных вызовов GCP даёт 2M против 1M у AWS и Azure
- Stateful-оркестрации — только у Azure (Durable Functions), у остальных нужны
  отдельные сервисы
- По количеству нативных триггеров AWS выигрывает — 200+ сервисов-источников
- Инструменты для локальной разработки и отладки более зрелые у AWS (SAM Local)

### Что выбрать для REST API backend

Для REST API backend я бы выбрала AWS Lambda + API Gateway. Основные аргументы:

Во-первых, Lambda — самый зрелый serverless-сервис с наибольшим комьюнити и
инструментарием. Есть SAM, Serverless Framework, CDK, хорошая локальная эмуляция
через SAM Local и нормальное distributed tracing через X-Ray. Это важно в продакшне,
где нужно быстро находить проблемы.

Во-вторых, Provisioned Concurrency решает проблему cold start без перехода на
always-on тариф. У Azure аналог — Premium за ~$150/мес постоянно, что дорого для
небольшого API. Lambda позволяет прогревать только нужные функции выборочно.

В-третьих, гибкость памяти от 128 МБ до 10 ГБ с пропорциональным CPU позволяет
тонко настраивать соотношение цены и производительности под каждый эндпоинт.

Если API сильно завязан на BigQuery или Pub/Sub — Cloud Functions 2nd gen интереснее
из-за 60-минутного таймаута и большего бесплатного уровня. Если нужны сложные
многошаговые воркфлоу с Azure AD — Azure Functions с Durable Functions.

### Плюсы и минусы serverless

Преимущества:
- Не нужно управлять серверами — нет патчинга ОС, настройки кластеров и планирования
  мощностей
- Автомасштабирование до нуля — при отсутствии трафика ничего не платишь
- Быстрый старт нового сервиса — деплой занимает секунды, не нужен Kubernetes
- Высокая доступность из коробки — функции запускаются в нескольких зонах доступности
- Детальный биллинг — оплата с точностью до миллисекунды

Недостатки:
- Cold start — функция, которую давно не вызывали, стартует медленно. Для API с
  требованиями по latency это критично и требует дополнительных решений
- Лимиты по времени выполнения — не подходит для долгих задач вроде обучения моделей
  или обработки больших видеофайлов без архитектурных обходов
- Vendor lock-in — триггеры, IAM и мониторинг у каждого провайдера свои; мигрировать
  между провайдерами дорого
- Сложность отладки — локальная эмуляция неточная, некоторые баги воспроизводятся
  только в облаке
- Stateless по природе — любое состояние нужно хранить во внешней БД или кеше, что
  усложняет архитектуру для задач с долгими сессиями
