## Задание 1

### Обзор и характеристики сервисов

**AWS — Amazon Elastic Container Registry (ECR)**
**Название сервиса:** Amazon ECR
**Характеристики:** Автоматическое сканирование образов на уязвимости, политики жизненного цикла для удаления старых версий, высокая доступность.
**Типы артефактов:** Docker-образы, OCI-образы, Helm-чарты.
**Интеграция:** Нативная работа с AWS ECS, EKS, Lambda, AWS CodeBuild и IAM.

**GCP — Artifact Registry**
**Название сервиса:** Google Cloud Artifact Registry
**Характеристики:** Поддержка региональных и мультирегиональных репозиториев, встроенный сканер безопасности, управление на основе ролей (IAM).
**Типы артефактов:** Docker, npm, Maven, Python (PyPI), NuGet, Go, пакеты ОС (Debian/RPM).
**Интеграция:** Глубокая интеграция с Cloud Build, GKE, Cloud Run и Google Cloud SDK.

**Azure — Azure Container Registry (ACR)**
**Название сервиса:** Azure Container Registry (ACR)
**Характеристики:** Гео-репликация (в тарифе Premium), доверие к контенту (Content Trust), автоматическая сборка образов (ACR Tasks).
**Типы артефактов:** Docker-образы, Helm-чарты, артефакты OCI.
**Интеграция:** Связка с Azure Kubernetes Service (AKS), Azure DevOps, GitHub и Azure Active Directory.

**Сходства:** Все три сервиса поддерживают стандарты OCI и Docker, предлагают сканирование на уязвимости и используют модель оплаты "по мере потребления" за хранение данных.
**Различия:** GCP предлагает самый широкий спектр типов артефактов (не только контейнеры); Azure выделяется лучшими возможностями глобальной репликации; AWS максимально оптимизирован для узкой работы с контейнерными оркестраторами.

**Анализ: Выбор для мультиоблачной стратегии**
Для мультиоблачной стратегии я бы выбрала GCP Artifact Registry. 
Его универсальность (поддержка множества языков программирования и типов пакетов в одном месте) позволяет использовать его как центральный хаб для всех команд разработки, независимо от того, в каком облаке развернута инфраструктура. Это значительно снижает сложность управления доступами и аудита.


## Задание 2

### Характеристики платформ

**AWS Lambda**
**Среды/Языки:** Node.js, Python, Java, Go, Ruby, .NET, Custom Runtimes.
**Цена:** $0.20 за 1 млн запросов + плата за время выполнения (ГБ-секунды).
**Производительность:** Среднее время холодного старта; высокая масштабируемость; лимит выполнения — 15 минут.
**Возможности:** Огромное количество триггеров (S3, SQS, Kinesis, DynamoDB).

**Google Cloud Functions**
**Среды/Языки:** Node.js, Python, Go, Java, PHP, Ruby, .NET.
**Цена:** Бесплатный уровень (2 млн вызовов) + плата за вызовы, процессор и память.
**Производительность:** Быстрый холодный старт для легких языков (Go/Python); лимит выполнения — 9 минут (HTTP).
**Возможности:** Отличная интеграция с Firebase и BigQuery; простая настройка HTTP-триггеров.

**Azure Functions**
**Среды/Языки:** .NET (лучшая поддержка), Node.js, Python, Java, PowerShell.
**Цена:** План Consumption (оплата за вызовы) или Premium (фиксированная плата за готовность).
**Производительность:** Гибкая; на плане Premium отсутствуют холодные старты; лимит выполнения до бесконечности (на спец. планах).
**Возможности:** Система Bindings (привязки), позволяющая подключать данные без написания SDK-кода.

### Сравнительный анализ

Я также не стала делать табличку, так как она смотрелась криво и нагружено. Мне показалось представить ответ в виде сходств и различий в две строчки - будет лучшим решением.

**Сходства:** Автоматическое масштабирование, отсутствие необходимости управлять серверами, поддержка основных языков программирования.
**Различия:** AWS обладает самой зрелой экосистемой; Azure предлагает уникальные возможности для .NET и бесконечного времени выполнения; GCP ориентирован на максимальную простоту развертывания.

**Анализ: Выбор для REST API бэкенда**
Для REST API я бы выбрала AWS Lambda.
Благодаря связке с AWS API Gateway, Lambda предоставляет самые мощные инструменты для управления API (дросселирование, версии, авторизация). Наличие функции Provisioned Concurrency позволяет полностью решить проблему задержек при "холодных стартах", что критично для отзывчивости интерфейса пользователя.

### Размышление: Плюсы и минусы Serverless

**Преимущества:**
1. **Экономия:** Нет оплаты за простаивающие ресурсы.
2. **Масштабируемость:** Система сама справляется с резкими скачками трафика.
3. **Фокус на коде:** DevOps-инженеры не тратят время на патчинг ОС и управление серверами.

**Недостатки:**
1. **Холодный старт:** Первая задержка при вызове функции.
2. **Ограничения времени:** Не подходит для длительной обработки данных (рендеринг, тяжелые вычисления).
3. **Мониторинг:** Сложнее отслеживать состояние распределенной системы по сравнению с монолитом.
