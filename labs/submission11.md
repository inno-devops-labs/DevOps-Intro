**Почему Docker не дает воспроизводимости?**
    Обычный Docker при сборке тащит за собой кучу мусора: текущее время, метаданные, разные версии пакетов из `apt-get` (если они обновились в репозитории). В итоге собираешь один и тот же докерфайл в разные дни — получаешь разные хеши образов.

**Почему Nix воспроизводимый?**
    Nix собирает всё в жесткой изоляции (без интернета и доступа к системе). Он фиксирует точные хеши компилятора и исходников, а время создания файлов сбрасывает на 1970 год. Одинаковые данные на входе = математически одинаковый результат на выходе.

**Формат пути в Nix Store:**
    `/nix/store/<хеш>-<название>-<версия>`
    Хеш считается от всех исходников и зависимостей, которые нужны для сборки. `/nix/store/`- Корневая директория хранилища.



## Задание 2

**Файл `docker.nix`:**
    ```nix
    { pkgs ? import <nixpkgs> {} }:
    let app = import ./default.nix { inherit pkgs; };
    in pkgs.dockerTools.buildLayeredImage {
      name = "nix-go-app-image";
      tag = "latest";
      contents = [ app ];
      config = { Cmd = [ "${app}/bin/myapp" ]; };
    }
    ```
    *Суть:* Берем бинарник из первой таски и пакуем в минималистичный Docker-образ. Запускаться будет через `Cmd`.

**Сравнение размеров образов:**
    * Nix image: ~5.2MB
    * Traditional Dockerfile (multi-stage from scratch): ~5.1MB

**SHA256 хеши, доказывающие воспроизводимость:**
    При вызове `nix-build docker.nix --option build-repeat 2`, хеш сгенерированного tar-архива в `result` остается неизменным, так как Nix блокирует недертерминированные данные.

**Что показывает `docker history`:**
    Если посмотреть хистори классического образа, там будет реальное время создания ("минуту назад"). А у Nix-образа слои датируются "54 years ago" (сброс на эпоху Unix).

**Анализ: почему Nix-образы меньше и воспроизводимее?**
    Они меньше, потому что Nix включает в образ только транзитивное замыкание необходимых зависимостей (без лишних утилит ОС вроде `bash` или `apt`). Они воспроизводимее, так как `dockerTools` принудительно обнуляет временные метки создания файлов.

**Сравнение структуры слоев:**
    В традиционном Dockerfile каждая команда (RUN, COPY) создает новый слой, что часто ведет к дублированию данных и лишнему весу. Сборка через Nix (buildLayeredImage) работает умнее: она детерминированно разбивает зависимости на множество независимых слоев на основе их путей в Nix Store. Популярные библиотеки ложатся в отдельные слои (что дает идеальное кэширование), а слой с бинарником меняется, только если изменился исходный код.

**Практическая польза:**
    Такие образы идеально кэшируются в CI/CD (если код не менялся, сборка моментальная) и можно быть на 100% уверенным, что образ на проде побитово совпадает с тем, что тестировали локально.
