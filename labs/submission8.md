### Анализ
В системе наблюдается умеренная нагрузка на CPU, при этом процесс chrome потребляет наибольший процент процессорного времени. Процесс node также вносит значительный вклад в загрузку CPU, что указывает на запущенный сервер или приложение. Использование памяти распределено аналогично: chrome и node занимают наибольшую долю, а docker потребляет дополнительную память из-за работы контейнеров. Дисковая активность в основном связана с процессом docker. Процесс systemd-journal также генерирует постоянную запись данных из-за ведения логов. Согласно выводу iostat, основной диск имеет умеренную загрузку (~38%) и приемлемое время ожидания, что говорит об отсутствии серьёзных узких мест Анализ использования диска показывает, что директория /var занимает значительный объём, в основном за счёт логов и данных docker. Крупные файлы, такие как syslog и kern.log, указывают на накопление логов без регулярной очистки.

### Рефлексия
Для оптимизации использования ресурсов необходимо настроить ротацию логов с помощью logrotate, чтобы предотвратить чрезмерный рост файлов в /var/log. Также рекомендуется очистить неиспользуемые Docker-образы и контейнеры для уменьшения занимаемого пространства. Нагрузку на CPU и память можно снизить за счёт оптимизации или ограничения ресурсоёмких процессов, таких как chrome и node, если они не являются критически важными. Дополнительно можно установить ограничения ресурсов для контейнеров Docker.Внедрение системы мониторинга и оповещений позволит своевременно выявлять аномалии в использовании ресурсов. В целом, предложенные меры помогут повысить стабильность системы и предотвратить исчерпание ресурсов.


## Task 2 — Практическая настройка мониторинга сайта

### URL сайта
https://innopolis.university

### Конфигурация Browser Check

Для мониторинга был создан browser check с использованием Checkly, который имитирует поведение реального пользователя.
Проверка выполняет следующие действия:
- Открывает сайт в браузере (Chromium)
- Ожидает полной загрузки страницы
- Проверяет наличие контента на странице (через body)
- Делает скриншот страницы
- Проверяет, что HTTP-статус ответа меньше 400
Проверка запускается каждые 5 минут из нескольких регионов.

### Результат успешного Browser Check

Browser check был успешно выполнен.

- Статус: Passed  
- Время выполнения: ~8–9 секунд  
- Страница загрузилась корректно  
- Ошибок не обнаружено  
- Скриншот был успешно сделан  
Это подтверждает, что сайт доступен и корректно отображается для пользователей.

### Настройка алертов

Для уведомлений были настроены индивидуальные (specific) параметры оповещений.

- Уведомление отправляется при **1 неудачной проверке**
- Канал уведомления: Email
- Включены типы уведомлений:
  - Ошибка (Failure)
  - Предупреждение (Warning)
  - Восстановление (Recovery)

Это позволяет оперативно узнавать о проблемах с сайтом.

### Анализ

Данная конфигурация мониторинга сочетает в себе два уровня проверки:

- **API check** проверяет доступность сервера и корректность ответа
- **Browser check** проверяет работу сайта с точки зрения пользователя
Такой подход позволяет обнаруживать:
- проблемы на сервере (например, недоступность API)
- проблемы интерфейса (например, страница не загрузилась или не отобразилась)
Запуск проверок из разных регионов также помогает выявлять региональные проблемы.

### Рефлексия
Настроенный мониторинг помогает повысить надежность сайта за счет быстрого обнаружения проблем.
В случае сбоя:
- система обнаружит проблему в течение нескольких минут
- будет отправлено уведомление
- можно быстро начать устранение проблемы
В реальных условиях такую систему можно улучшить:
- добавить проверки пользовательских сценариев (например, вход в систему)
- мониторить критически важные страницы
- подключить дополнительные каналы уведомлений (Slack, Telegram)
В целом, данный мониторинг обеспечивает базовый уровень надежности и помогает поддерживать стабильную работу сайта.
